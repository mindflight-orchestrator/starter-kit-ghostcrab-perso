# DDL Projection

A DDL Projection is a pre-assembled context cockpit. It is a concept in the mindBrain model, not merely a query result.

In GhostCrab MCP, the corresponding agent-facing tools are `ghostcrab_project` and `ghostcrab_pack`. They expose compact working projections and context packs; they do not guarantee the exact rich JSON shape shown in this article.

---

## DDL, not SQL DDL

In SQL, DDL means Data Definition Language: `CREATE TABLE`, `ALTER TABLE`, and related schema operations.

In mindBrain, DDL means Domain Definition Language: a domain model that declares entities, facets, relations, and projections. A projection says what context should be assembled for an agent, which graph relations matter, and what state should be surfaced at runtime.

Same acronym. Different intent.

---

## The problem

Every time an agent starts a task, it needs context:

1. current entity state
2. relevant facts
3. nearby graph relations
4. blockers and constraints
5. likely next steps

Without projections, the agent reconstructs this on every turn through repeated reads and inference. That costs latency, tokens, and reliability.

Projection moves context assembly toward the memory layer.

---

## Engine-level cockpit shape

A rich projection cockpit can look like this:

```json
{
  "position": {
    "entity_type": "Task",
    "id": "task-gdpr-art32",
    "status": "blocked",
    "priority": "critical",
    "domain": "compliance"
  },
  "available_moves": [
    {
      "relation": "assigned_to",
      "target": "agent-legal-01",
      "status": "available"
    },
    {
      "relation": "blocks",
      "target": "task-gdpr-art35",
      "status": "planned"
    }
  ],
  "constraints": [
    {
      "rule": "blocked_requires_owner",
      "severity": "error",
      "message": "Blocked task must have an assigned agent."
    }
  ],
  "context_window": {
    "summary": "Task blocked since 2026-04-28. GDPR Art. 32 requires 4 controls. 2 validated, 2 pending audit.",
    "depth": 2,
    "included_entities": ["Regulation:gdpr-art32", "Control:ctrl-01", "Control:ctrl-02"]
  },
  "next_options": [
    "Assign an available agent to unblock.",
    "Escalate if no agent is available within 24h."
  ]
}
```

This is illustrative. It describes the target shape of an engine-level cockpit. A live MCP `ghostcrab_pack` response is a compact working-memory pack assembled from active projections and matching facts; clients should treat the live response as the contract.

## MCP projection tools

Use `ghostcrab_project` to create or refresh a compact projection:

```json
{
  "tool": "ghostcrab_project",
  "arguments": {
    "scope": "compliance-overview",
    "content": "Track GDPR Art. 32 controls, blockers, owners, and audit evidence gaps.",
    "proj_type": "GOAL",
    "status": "active",
    "agent_id": "agent:compliance",
    "provisional": false
  }
}
```

`proj_type` is one of:

- `FACT`
- `GOAL`
- `STEP`
- `CONSTRAINT`

Use `ghostcrab_pack` to read compact context:

```json
{
  "tool": "ghostcrab_pack",
  "arguments": {
    "workspace_id": "compliance",
    "scope": "compliance-overview",
    "query": "current blockers and pending evidence for GDPR Art. 32",
    "agent_id": "agent:compliance",
    "limit": 15
  }
}
```

For fuzzy first-turn onboarding, do not create projections immediately. Model the domain first, ask the necessary clarification questions, and write only after confirmation.

## Pro and `pg_pragma`

In GhostCrab Pro, `pg_pragma` can accelerate packed context retrieval when the runtime reports readiness. Projection creation still goes through normal persistence via the MCP tool or the operator path.

Check runtime reality with:

```json
{
  "tool": "ghostcrab_status",
  "arguments": {
    "agent_id": "agent:compliance"
  }
}
```

Relevant Pro readiness fields include `runtime.extensions_detected.pg_pragma`, `runtime.native_readiness.pragma`, and `runtime.capabilities.pragma_native_pack`.

## Cached vs on-demand

| Mode | Meaning | Agent impact |
|---|---|---|
| Engine-level cached projection | Context assembled by the memory/runtime layer | Fast, stable cockpit when supported |
| MCP pack | Compact pack assembled from active projections and matching facts | Practical agent-facing read path |
| Agent self-assembly | Agent manually searches, counts, traverses, and summarizes | Flexible but slower and easier to get wrong |

Use projections for repeated work: active blockers, compliance coverage, release state, incident status, and weekly focus.

---

## How projections relate to graph and facets

A useful projection usually combines:

- faceted state, such as `status`, `priority`, `owner`, `domain`
- graph relations, such as `blocks`, `requires`, `validated_by`
- constraints or goals, such as "no critical blocker without owner"
- a compact text summary for the agent

Facets answer "where is this thing in state space?" Graph answers "what is it connected to?" Projection answers "what does the agent need now?"
