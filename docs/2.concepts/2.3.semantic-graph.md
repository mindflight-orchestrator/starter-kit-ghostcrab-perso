# Semantic Graph

Relationships are first-class objects: typed, directional, and traversable.

In mindBrain, graph semantics explain what edges mean. In GhostCrab MCP, agents write graph structure with `ghostcrab_learn`, traverse it with `ghostcrab_traverse`, and check domain fit with `ghostcrab_coverage`.

---

## Why foreign keys are not enough

A relational database can store that `task_id = 42` has `assigned_agent_id = 7`. That link is useful, but it does not tell an agent what the relation means.

`blocks`, `depends_on`, and `requires` are all directed edges. They have different semantics:

- **`blocks`**: active obstruction; one item cannot advance until another resolves.
- **`depends_on`**: ordered dependency; one item should happen after another.
- **`requires`**: normative obligation; one thing requires another thing to exist or reach a state.

An agent that only sees ids must infer meaning. A semantic graph stores meaning in the edge label.

---

## Declaring and confirming relations

At the concept level, relations can be described in a domain model:

```yaml
entities:
  Task:
    relations:
      - name: blocks
        target: Task
        inverse_name: blocked_by

      - name: assigned_to
        target: Agent
        inverse_name: assignments

  Regulation:
    relations:
      - name: requires
        target: Control
        inverse_name: required_by
```

Do not assume every runtime automatically creates inverse traversal from YAML. In GhostCrab, schema and ontology behavior depends on the confirmed model, registered schemas, and runtime capabilities. Document inverses as model intent unless the active system has registered them.

## Writing graph structure with MCP

Use `ghostcrab_learn` for durable graph nodes and edges. It is an extended tool, so discover it if it is not visible by default:

```json
{
  "tool": "ghostcrab_tool_search",
  "arguments": {
    "name_prefix": "ghostcrab_learn",
    "include_input_schema": true,
    "limit": 5
  }
}
```

Write an edge:

```json
{
  "tool": "ghostcrab_learn",
  "arguments": {
    "workspace_id": "compliance",
    "edge": {
      "source": "regulation:gdpr-art32",
      "target": "control:encryption-at-rest",
      "label": "requires",
      "weight": 1,
      "properties": {
        "article": "Art. 32"
      }
    }
  }
}
```

Optionally write a node:

```json
{
  "tool": "ghostcrab_learn",
  "arguments": {
    "workspace_id": "compliance",
    "node": {
      "id": "control:encryption-at-rest",
      "node_type": "Control",
      "label": "Encryption at rest",
      "properties": {
        "domain": "security"
      }
    }
  }
}
```

Do not create graph structure on a first fuzzy onboarding turn. Clarify and confirm the model first.

## Traversing the graph

Use `ghostcrab_traverse` for graph navigation:

```json
{
  "tool": "ghostcrab_traverse",
  "arguments": {
    "workspace_id": "compliance",
    "start": "regulation:gdpr-art32",
    "edge_labels": ["requires"],
    "direction": "outbound",
    "depth": 2
  }
}
```

To inspect the live schema:

```json
{
  "tool": "ghostcrab_tool_search",
  "arguments": {
    "name_prefix": "ghostcrab_traverse",
    "include_input_schema": true,
    "limit": 5
  }
}
```

In Pro, supported traversal cases can route through `pg_dgraph` when readiness passes. Other cases may use SQL fallback. In Perso, traversal availability and backend path depend on the live runtime.

## Coverage

Coverage asks whether the graph and ontology contain enough structure for a domain.

```json
{
  "tool": "ghostcrab_coverage",
  "arguments": {
    "domain": "compliance",
    "workspace_id": "compliance",
    "agent_id": "agent:compliance"
  }
}
```

In Pro, coverage can benefit from graph/ontology native readiness and confidence scoring. A gap result does not mean the domain is empty; it means the current graph does not fully cover the expected ontology.

## Facets plus graph

Facets answer "which records match this state?"

```json
{
  "tool": "ghostcrab_search",
  "arguments": {
    "workspace_id": "compliance",
    "schema_id": "ghostcrab:task",
    "filters": {
      "status": "blocked",
      "priority": "critical"
    },
    "limit": 20
  }
}
```

Graph traversal answers "what is connected to this record?"

Together, they support operational questions such as:

- Which critical tasks are blocked?
- Who owns them?
- What dependency is blocking each one?
- Which regulations require controls that are not yet validated?

## Graph feeds projections

Projections often include graph state. A compliance overview might combine:

- `Regulation requires Control`
- `Control validated_by Audit`
- `Control.status`
- pending evidence gaps

The agent-facing path is:

```json
{
  "tool": "ghostcrab_pack",
  "arguments": {
    "workspace_id": "compliance",
    "scope": "compliance-overview",
    "query": "GDPR Art. 32 controls, validation status, and evidence gaps",
    "agent_id": "agent:compliance"
  }
}
```

If a projection needs to be created or refreshed:

```json
{
  "tool": "ghostcrab_project",
  "arguments": {
    "scope": "compliance-overview",
    "content": "Track GDPR controls, required audits, validation state, and blockers.",
    "proj_type": "GOAL",
    "status": "active",
    "agent_id": "agent:compliance"
  }
}
```

## Operator path

For Pro batch or governed workflows, mindCLI can run approved query templates instead of asking an agent to assemble graph state:

```bash
mindcli pg query \
  --template gdpr_controls_with_audits \
  --workspace compliance \
  --param regulation_id=gdpr-art32 \
  --format json
```

Use this path for deterministic reports, scheduled checks, or high-volume jobs. Use MCP for agent-facing contextual reads and writes.
