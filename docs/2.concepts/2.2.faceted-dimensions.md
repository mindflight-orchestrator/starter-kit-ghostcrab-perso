# Faceted Dimensions

Objects exist at typed coordinates, not in a flat tag cloud.

In mindBrain, dimensions are the conceptual model. In GhostCrab MCP, the practical agent-facing tools are `ghostcrab_search` and `ghostcrab_count`.

---

## Dimensions vs tags

Tags search. Dimensions navigate.

A tag system answers: "find everything tagged `compliance`." You get a list and filter it yourself.

A dimension system answers: "how many compliance tasks are blocked, grouped by priority?" The memory layer can count by typed coordinates instead of asking the agent to reconstruct the answer.

The difference is architectural. Tags are annotations. Dimensions are coordinates in a modeled space.

---

## Common dimension types

| Type | Use for | Example values |
|---|---|---|
| `enum` | Controlled states and categories | `active`, `blocked`, `done` |
| `string` | Names, ids, labels, codes | `GDPR Art. 32`, `team-alpha` |
| `integer` | Counts and ranks | `0`, `42` |
| `float` | Scores, weights, percentages | `0.87`, `3.14` |
| `date` | Calendar dates | `2026-05-01` |
| `datetime` | Timestamps | `2026-05-01T09:00:00Z` |
| `boolean` | Binary flags | `true`, `false` |
| `json` | Flexible metadata | `{"source":"odoo"}` |
| `vector` | Semantic proximity when enabled | embedding arrays |

Default to the strictest type the data supports. Use `enum` for known values. Use `string` for genuinely open-ended identifiers. Use `vector` only when similarity search is part of the product behavior.

## Querying by facet with MCP

For agents, use `ghostcrab_search` with exact facet filters:

```json
{
  "tool": "ghostcrab_search",
  "arguments": {
    "workspace_id": "compliance",
    "schema_id": "ghostcrab:task",
    "filters": {
      "status": "blocked",
      "domain": "compliance"
    },
    "query": "",
    "mode": "bm25",
    "limit": 20
  }
}
```

`mode` can be `hybrid`, `bm25`, or `semantic`, depending on runtime capability and the kind of query. Empty `query` means pure facet filtering.

Arrays express OR within a facet:

```json
{
  "tool": "ghostcrab_search",
  "arguments": {
    "workspace_id": "compliance",
    "schema_id": "ghostcrab:task",
    "filters": {
      "status": ["blocked", "in_progress"],
      "domain": "compliance"
    },
    "limit": 20
  }
}
```

## Counting by dimension

Use `ghostcrab_count` when you need aggregation without loading full content. Its `group_by` argument is required.

```json
{
  "tool": "ghostcrab_count",
  "arguments": {
    "workspace_id": "compliance",
    "schema_id": "ghostcrab:task",
    "filters": {
      "status": "blocked"
    },
    "group_by": ["priority"]
  }
}
```

A second grouping dimension creates a compact cross-section:

```json
{
  "tool": "ghostcrab_count",
  "arguments": {
    "workspace_id": "compliance",
    "schema_id": "ghostcrab:task",
    "filters": {
      "domain": "compliance"
    },
    "group_by": ["status", "priority"]
  }
}
```

The exact response shape is runtime-defined. Treat the live tool result as the contract.

## Operator and Pro paths

For Perso setup and schema profile discovery, use `gcp brain`:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain schema list
```

For Pro deterministic reporting, use mindCLI or approved query templates:

```bash
mindcli pg query \
  --template blocked_tasks_by_priority \
  --workspace compliance \
  --format json
```

Do not use a generic legacy query command in this starter kit unless a distribution explicitly provides it.

## Vector and hybrid search

Vector and hybrid search are optional. They depend on embedding configuration and runtime readiness.

Use exact facets when:

- values are finite and known
- counts and filters must be deterministic
- the task is operational state, not semantic similarity

Use semantic or hybrid search when:

- the content is free-form
- similarity is the desired behavior
- `ghostcrab_status` reports embeddings/vector readiness

Perso may run BM25-only. Pro may expose stronger hybrid/native paths when `pg_facets`, embeddings, and readiness checks are available.

## Performance guidance

Performance depends on backend, index readiness, facet cardinality, query shape, and data volume. Fixed latency tables are only illustrative.

| Backend | Best fit | Notes |
|---|---|---|
| SQLite Perso | Local agents, personal workflows, starter projects | Good default for local state and small to medium graphs |
| PostgreSQL Pro | Concurrent agents, governed workspaces, larger datasets | Can use `pg_facets` when readiness passes |
| Pro plus mindCLI | Batch ingestion and repeatable reporting | Better than repeated MCP calls for high-volume operations |

Use `ghostcrab_status` for actual runtime capability.

---

## Modeling tips

- Put day-to-day state in facets, not only in prose.
- Prefer stable `record_id` values for upsert matching.
- Use enums for workflow state.
- Use graph edges for real relations such as `blocks`, `requires`, and `assigned_to`.
- Add vector dimensions only when semantic similarity is a real requirement.
