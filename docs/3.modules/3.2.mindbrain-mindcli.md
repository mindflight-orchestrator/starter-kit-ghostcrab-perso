# mindCLI

mindCLI is the deterministic operator channel for the Pro mindBrain stack.

It is not the same thing as the Perso `gcp brain` CLI. In this starter kit:

- use `gcp brain ...` to start and configure GhostCrab Perso;
- use GhostCrab MCP tools when an agent needs structured memory;
- use `mindcli ...` for Pro workflows that must run without LLM ambiguity, such as migrations, batch ingestion, sync, and query templates.

---

## Role

GhostCrab MCP is the contextual decision surface for agents. mindCLI is the deterministic execution surface for operators and automation.

| Need | GhostCrab MCP | mindCLI |
|---|---|---|
| Trigger | LLM agent or MCP client | Human operator, cron, CI/CD, webhook |
| Data volume | Small to medium tool calls | Batch and streaming workloads |
| Governance | Human confirmation inside agent workflow | Explicit approve/execute commands |
| SQL safety | Tool schemas and server validation | Registered templates and controlled operator commands |
| Output | MCP tool result JSON | stdout, structured logs, exit codes |

This split matters. High-volume ingestion should not be implemented as thousands of MCP calls. MCP is for ontology, query, context, and agent-safe operations. mindCLI is for deterministic operational work.

---

## Why not raw SQL

Raw SQL is useful for database administrators. It is not the right default interface for agents or scheduled business operations.

| Operation | Raw SQL problem | mindCLI shape |
|---|---|---|
| Create a workspace | Must know internal tables and schemas | `mindcli workspace create --id agency_projects --label "Agency Projects"` |
| Approve a migration | Easy to bypass governance | `mindcli migration approve --id <migration_uuid> --by "operator"` |
| Ingest an external source | Must hand-roll mapping and sync | `mindcli odoo ingest --workspace agency_projects --entity project.task` |
| Run approved reporting SQL | Raw query may leak or drift | `mindcli pg query --template task_by_phase --workspace agency_projects` |
| Repair Layer 1 to Layer 2 state | Requires precise table knowledge | `mindcli resync --workspace agency_projects --table project_task` |

mindCLI preserves the operational contract while staying scriptable.

## Command families

The Pro roadmap validates these command families as the intended operator surface.

### Workspace

```bash
mindcli workspace list --format table
mindcli workspace create --id agency_projects --label "Agency Projects"
mindcli workspace status --id agency_projects
```

Workspace commands mirror the governed workspace model used by MCP tools such as `ghostcrab_workspace_create`, `ghostcrab_workspace_list`, and `ghostcrab_workspace_inspect`.

### Migration

```bash
mindcli migration list --status pending
mindcli migration approve --id <migration_uuid> --by "operator"
mindcli migration reject --id <migration_uuid> --reason "schema needs review"
mindcli migration preview-trigger --id <migration_uuid>
mindcli migration execute --id <migration_uuid>
```

Agents can propose DDL through MCP. Operators approve and execute through mindCLI when the workflow requires human governance.

### Ingest

```bash
mindcli odoo ingest \
  --workspace agency_projects \
  --entity project.task \
  --since "2026-01-01" \
  --batch-size 500
```

The ingest path writes to Layer 1 tables and lets sync/triggers update Layer 2 facets and graph structures. It is the right channel for bulk sources.

### Query templates

```bash
mindcli pg query \
  --template task_by_phase \
  --workspace agency_projects \
  --param phase_id=42 \
  --format json
```

```bash
mindcli pg query --list --workspace agency_projects
mindcli pg query --register \
  --id task_by_phase \
  --workspace agency_projects \
  --file query_task_by_phase.sql \
  --description "Tasks by phase and status"
```

Templates let humans register approved SQL shapes without letting agents generate arbitrary database access.

### Sync and resync

```bash
mindcli sync plan --file sync_plan.yaml --workspace agency_projects
mindcli sync run --plan agency_projects_daily
mindcli sync status --plan agency_projects_daily
```

```bash
mindcli resync --workspace agency_projects --table project_task
```

Sync commands are for repeatable multi-source data flows. Resync is for repair and reconciliation.

## Relationship to Perso `gcp brain`

Perso uses `gcp brain` for local setup:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain workspace create my-app
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain up --workspace my-app
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain schema list
```

Those commands are not mindCLI. They start and configure the local Perso runtime.

When a doc says `mindcli`, read it as Pro/operator scope unless the distribution explicitly says otherwise.

## Output and scripting

mindCLI commands should be friendly to scripts:

- JSON output for machine parsing
- table output for humans
- non-zero exit codes on validation, conflict, not-found, or backend errors
- structured error messages that identify the workspace, migration, template, or source involved

Example:

```bash
PENDING=$(mindcli migration list --status pending --format json)
echo "$PENDING" | jq -r '.[].id'
```

## Where agents fit

Agents should still use MCP for contextual reads and writes:

```json
{
  "tool": "ghostcrab_pack",
  "arguments": {
    "workspace_id": "agency_projects",
    "query": "current blocked tasks and active migration context",
    "agent_id": "agent:ops"
  }
}
```

When the agent reaches a governed operation, it should produce or route to the corresponding mindCLI command instead of bypassing the workflow.
