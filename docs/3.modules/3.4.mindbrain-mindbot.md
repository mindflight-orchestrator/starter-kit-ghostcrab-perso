# mindBot

mindBot is an advanced workflow conductor for mindBrain-backed systems.

It is not the default Perso runtime. In this documentation set, treat mindBot as a Pro or roadmap-level orchestration layer that coordinates agents, roles, guards, and handoffs using structured state stored in mindBrain.

---

## The problem without it

Independent agents can share a database and still fail to coordinate.

In a compliance audit, for example:

- a researcher maps regulations to controls;
- a validator checks controls against evidence;
- a reporter assembles the audit trail.

Without a conductor, the reporter can run before validation is complete, a validator can miss pending controls, and the workflow state may exist only in conversation logs.

mindBot solves that at the workflow layer. It reads mindBrain state, identifies the active step, routes the right role, and checks handoff conditions before advancing.

---

## What mindBot adds

- **Role assignment**: bind agent identities or humans to declared workflow roles.
- **SOP definition**: represent repeatable procedures as ordered steps with entry and exit conditions.
- **Workflow routing**: decide which role should act next from graph and facet state.
- **Handoff guards**: advance only when mindBrain state satisfies the guard.
- **Auditability**: keep workflow state in structured records and graph relations, not only logs.

mindBot decides what should happen next. LangGraph, CrewAI, Codex, Cursor, Claude, or a human operator can still decide how a role is executed.

---

## Core concepts

**Workflow**
A named process with ordered steps, a root entity, and terminal states.

**Role**
An execution responsibility, such as researcher, validator, reporter, operator, or reviewer.

**SOP step**
A workflow step with a role, entry guard, exit condition, and optional timeout/escalation behavior.

**Guard**
A condition evaluated against mindBrain state. Guards should be expressed as facet filters, graph traversals, coverage checks, or approved query templates.

**Handoff**
The transition from one SOP step to another after the exit condition passes.

## Example workflow

```yaml
workflows:
  compliance_audit:
    root_entity: Regulation
    roles:
      - researcher
      - validator
      - reporter

    steps:
      - id: map_controls
        role: researcher
        entry_guard:
          regulation.status: pending
        exit_condition:
          graph.requires_controls: exists
        on_exit:
          set:
            regulation.status: mapped
          advance_to: validate_controls

      - id: validate_controls
        role: validator
        entry_guard:
          regulation.status: mapped
        exit_condition:
          controls.pending_count: 0
        on_exit:
          set:
            regulation.status: validated
          advance_to: produce_report

      - id: produce_report
        role: reporter
        entry_guard:
          regulation.status: validated
        exit_condition:
          audit.report_exists: true
        on_exit:
          set:
            workflow.status: complete
```

This YAML is illustrative. A production implementation should bind each guard to either MCP reads, `mindcli pg query` templates, or an internal workflow table.

## Routing sequence

At the start of a workflow turn, mindBot can combine deterministic reads and agent-safe MCP tools:

```bash
mindcli pg query \
  --template workflow_state \
  --workspace compliance \
  --param regulation_id=gdpr-art32 \
  --format json
```

```json
{
  "tool": "ghostcrab_pack",
  "arguments": {
    "workspace_id": "compliance",
    "scope": "validator-cockpit",
    "query": "pending controls and evidence gaps for GDPR Art. 32",
    "agent_id": "agent:validator"
  }
}
```

The sequence is:

1. Read workflow state.
2. Identify the active step.
3. Evaluate entry guard.
4. Load the role projection or pack.
5. Dispatch the role to a harness or human.
6. Read the result from mindBrain.
7. Evaluate exit condition.
8. Advance or escalate.

The agent does not decide the next step by vibe. The workflow advances when structured state says it can.

## Shared state, not direct chatter

Agents in a mindBot workflow should not need private message passing.

```text
Researcher -> GhostCrab MCP / mindCLI -> mindBrain state
Validator  -> GhostCrab MCP / mindCLI -> mindBrain state
Reporter   -> GhostCrab MCP / mindCLI -> mindBrain state
mindBot    -> reads state, routes roles, enforces guards
```

That design makes agents replaceable. A role can be fulfilled by a model, a human, a scheduled job, or a different harness without rewriting the workflow.

---

## mindBot vs harnesses

| | LangGraph / CrewAI / IDE agent | mindBot |
|---|---|---|
| Primary job | Execute agent code and tool calls | Decide workflow state and handoffs |
| State | In-memory or harness-managed | mindBrain records, graph, and projections |
| Conditions | Code-level branches | Declarative guards over structured state |
| Interface to memory | MCP tools or SDK calls | MCP reads plus mindCLI/operator reads |
| Audit trail | Logs and traces | Queryable structured state |

Typical advanced stack:

```text
mindBot
  -> LangGraph / CrewAI / IDE agent / human role
      -> GhostCrab MCP for contextual memory
      -> mindCLI for deterministic Pro operations
          -> mindBrain state
```

## Scope in this starter kit

Perso users do not need mindBot to get value from GhostCrab. Start with MCP tools, workspace setup, and projections.

Use mindBot when workflows need governed handoffs, multiple roles, audit trails, and deterministic state transitions over a Pro/mindCLI-backed system.
