# GhostCrab MCP

GhostCrab MCP exposes mindBrain as typed tools for MCP-compatible clients such as Cursor, Codex, Claude, and other agent harnesses.

MCP tools are declared with names, descriptions, and JSON input schemas. The MCP specification describes this contract in the tools section: https://modelcontextprotocol.io/specification/2025-11-25/server/tools

---

## What GhostCrab adds to MCP

MCP gives agents a standard way to call external tools. GhostCrab adds a durable structured memory backend:

- faceted records for current state and notes
- graph relations for blockers, dependencies, and coverage
- projections for resumable working context
- workspace and ontology semantics where the product surface supports them

An MCP client should not infer tool payloads from prose. It should inspect the live catalog and call tools with JSON arguments that match `inputSchema`.

```json
{
  "tool": "ghostcrab_tool_search",
  "arguments": {
    "name_prefix": "ghostcrab_workspace_",
    "include_input_schema": true,
    "limit": 10
  }
}
```

## Perso vs Pro

GhostCrab has multiple product surfaces. They share concepts, but not identical scope.

| | GhostCrab Perso | GhostCrab Pro / mindBrain PostgreSQL |
|---|---|---|
| Package | `@mindflight/ghostcrab-personal-mcp` | `@mindflight/ghostcrab` |
| Runtime | SQLite-backed local MindBrain | PostgreSQL native stack |
| CLI | `gcp brain ...` | Pro package plus `mindcli` operator workflows |
| Store | `./data/ghostcrab.sqlite` by default | PostgreSQL database |
| MCP surface | Compact default list plus discoverable catalog | Broader native/catalog surface |
| Native readiness | SQLite path, optional local capabilities | `pg_facets`, `pg_dgraph`, `pg_pragma`, `pg_mindbrain` readiness |
| Best for | Local agents, starter kits, individual workflows | Team deployments, governed workspaces, scale, concurrent agents |

Do not document a fixed "same N tools" promise across products. Tool count depends on package, host, enabled catalog, extensions, and client limits.

## Install and start Perso

For this starter kit, Perso is the local path:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp --help
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain workspace create my-app
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain up --workspace my-app
```

Perso creates and uses a local SQLite store such as `./data/ghostcrab.sqlite`, depending on current working directory and environment.

To wire an IDE, prefer the generated setup commands documented in the installation guide:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain setup cursor --force
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain setup codex
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain setup claude
```

## Pro scope

GhostCrab Pro is PostgreSQL-backed and can expose a broader native and ontology surface. The Pro repo validates:

- native PostgreSQL boot with `pg_facets`, `pg_dgraph`, `pg_pragma`
- `pg_mindbrain` for unified ontology/workspace capabilities when loaded
- workspace semantics, DDL proposal/list/execute lifecycle, and model export
- graph marketplace, patch, traversal, and coverage capabilities when readiness checks pass
- optional geo queries when PostGIS and geo migrations are available

The local Pro MCP server may be unavailable if PostgreSQL is down. That is an environment condition, not evidence that Pro lacks those services. Use the Pro repo architecture docs and `ghostcrab_status` from a healthy Pro instance for final runtime truth.

## Tool surfaces

Perso typically exposes a compact default set to the agent and a larger catalog through `ghostcrab_tool_search`. Pro exposes a larger native/catalog surface when its backend and extensions are ready.

Useful groups:

| Group | Typical tools |
|---|---|
| Facets | `ghostcrab_search`, `ghostcrab_remember`, `ghostcrab_upsert`, `ghostcrab_count`, `ghostcrab_facet_tree`, `ghostcrab_schema_list`, `ghostcrab_schema_inspect`, `ghostcrab_schema_register` |
| Graph | `ghostcrab_learn`, `ghostcrab_traverse`, `ghostcrab_marketplace`, `ghostcrab_patch`, `ghostcrab_coverage` |
| Projections | `ghostcrab_project`, `ghostcrab_pack`, `ghostcrab_status` |
| Workspace / DDL | `ghostcrab_workspace_create`, `ghostcrab_workspace_list`, `ghostcrab_workspace_inspect`, `ghostcrab_workspace_export_model`, `ghostcrab_ddl_propose`, `ghostcrab_ddl_list_pending`, `ghostcrab_ddl_execute` |
| Pro ontology/loadout | loadout, ontology compare, bridge, conflicts, coverage reference, batch/project-template tools where enabled |
| Optional geo | `ghostcrab_query_geo` when the backend supports geo entities |

This table is explanatory, not a schema contract. Use `ghostcrab_tool_search` for the exact live shape.

## Valid payload examples

Create a workspace:

```json
{
  "tool": "ghostcrab_workspace_create",
  "arguments": {
    "id": "agency-projects",
    "label": "Agency Projects",
    "description": "Workspace for agency delivery projects."
  }
}
```

Register a schema:

```json
{
  "tool": "ghostcrab_schema_register",
  "arguments": {
    "workspace_id": "agency-projects",
    "target": "facets",
    "definition": {
      "schema_id": "agency:task",
      "description": "A task tracked in an agency delivery workflow.",
      "version": 1,
      "facets": {
        "record_id": "string",
        "status": "enum:todo|in_progress|blocked|done",
        "owner": "string"
      }
    }
  }
}
```

Update a current-state record:

```json
{
  "tool": "ghostcrab_upsert",
  "arguments": {
    "workspace_id": "agency-projects",
    "schema_id": "agency:task",
    "match": {
      "facets": {
        "record_id": "task:launch-review"
      }
    },
    "set_content": "Launch review is blocked on final compliance evidence.",
    "set_facets": {
      "record_id": "task:launch-review",
      "status": "blocked",
      "owner": "legal"
    },
    "create_if_missing": true
  }
}
```

Check coverage:

```json
{
  "tool": "ghostcrab_coverage",
  "arguments": {
    "domain": "agency-delivery",
    "workspace_id": "agency-projects",
    "agent_id": "agent:delivery"
  }
}
```

Create a projection:

```json
{
  "tool": "ghostcrab_project",
  "arguments": {
    "scope": "agency-delivery-overview",
    "content": "Track blocked launch tasks, owners, and compliance evidence gaps.",
    "proj_type": "GOAL",
    "status": "active",
    "agent_id": "agent:delivery"
  }
}
```

## First-turn intake

For fuzzy first-turn onboarding, follow the GhostCrab agent contract:

- read and model before writing;
- use `ghostcrab_modeling_guidance` when the user asks to structure a domain;
- ask 2 to 4 high-impact clarification questions;
- write only after model confirmation;
- treat the five-question ontology checklist as teaching material, not a mandatory first response.

This avoids contradicting the live server instructions agents actually receive.
