# mindBrain Core

mindBrain is the structured knowledge engine behind GhostCrab. It gives agents and operators a shared model for entities, facets, relations, projections, and workspace-scoped semantics.

The important distinction is product surface:

- **GhostCrab Perso** uses a local SQLite-backed MindBrain runtime and the `gcp brain` CLI.
- **GhostCrab Pro** uses PostgreSQL with native extensions and a broader MCP/catalog surface.
- **mindCLI** is the deterministic operator channel for Pro workflows such as migration approval, batch ingestion, sync, and query templates.

The model is shared. The available commands and runtime guarantees depend on the product layer.

---

## What the core does

mindBrain Core has four responsibilities:

1. **Store typed facts and entities** with stable ids, schema ids, facets, and content.
2. **Index dimensions** so status, priority, owner, domain, and other facets are queryable.
3. **Maintain graph relations** for blockers, dependencies, coverage, and conceptual links.
4. **Assemble projections** so an agent can resume a compact working context instead of rereading history.

Everything else is an interface on top:

- MCP tools for agents: `ghostcrab_search`, `ghostcrab_upsert`, `ghostcrab_traverse`, `ghostcrab_pack`
- Perso CLI for local runtime setup: `gcp brain ...`
- Pro mindCLI for deterministic operations: `mindcli ...`
- Harnesses and workflow layers: Cursor, Codex, Claude, LangGraph, CrewAI, mindBot

## Architecture

```text
User / Operator / Agent
        |
        |-- Perso setup:  gcp brain ...
        |-- Pro ops:      mindcli ...
        |-- Agent tools:  ghostcrab_* MCP tools
        v
mindBrain model
  - schemas and ontology profiles
  - facets and current-state records
  - graph nodes and edges
  - projections and working packs
        |
        |-- Perso: SQLite local runtime
        |-- Pro:   PostgreSQL native stack
        v
Storage and native acceleration
```

In Perso, the default local store is a SQLite file such as `./data/ghostcrab.sqlite`, managed through `gcp brain up`.

In Pro, PostgreSQL can use native extension readiness from `pg_facets`, `pg_dgraph`, `pg_pragma`, and `pg_mindbrain`. Extension presence alone is not enough; `ghostcrab_status` reports the active readiness and backend routing.

## Two backends, scoped APIs

The backend does not change the basic domain concepts, but it does change scope and operational coverage.

| Layer | Perso | Pro / PostgreSQL |
|---|---|---|
| Runtime | SQLite local backend | PostgreSQL native stack |
| Setup CLI | `gcp brain ...` | Pro package and Docker/native boot |
| Agent interface | GhostCrab MCP compact default surface plus discoverable catalog | GhostCrab MCP broader native/catalog surface |
| Operator interface | Local workspace/profile commands | `mindcli` for migrations, ingest, sync, query templates |
| Best fit | Local agents, starter kits, individual workflows | Team deployments, concurrent agents, large datasets, governed data flows |

Do not assume one flat command set exists everywhere. For example:

```bash
# Perso runtime setup
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain workspace create my-app
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain up --workspace my-app
```

```bash
# Pro deterministic operator workflow
mindcli workspace create --id agency_projects --label "Agency Projects"
mindcli migration list --status pending
mindcli migration approve --id <migration_uuid> --by "operator"
```

```json
{
  "tool": "ghostcrab_search",
  "arguments": {
    "workspace_id": "agency_projects",
    "schema_id": "ghostcrab:task",
    "filters": {
      "status": "blocked"
    },
    "limit": 20
  }
}
```

## What lives in the database

| Structure | Contents | Typical interface |
|---|---|---|
| Facets / records | Content plus typed dimensions such as status, owner, priority, domain | `ghostcrab_remember`, `ghostcrab_upsert`, `ghostcrab_search`, `ghostcrab_count` |
| Schema / ontology registry | Schema ids, facet contracts, node types, edge labels, ontology loadouts | `ghostcrab_schema_*`, `gcp brain schema`, Pro ontology tools |
| Graph | Directed typed edges and node metadata | `ghostcrab_learn`, `ghostcrab_traverse`, `ghostcrab_coverage`, graph marketplace tools |
| Pro workspace semantics | Layer 1 table semantics, DDL proposals, trigger sync specs, exports | Workspace and DDL MCP tools, `mindcli migration`, `mindcli pg query` |
| Projections | Compact working contexts for agents | `ghostcrab_project`, `ghostcrab_pack` |

Perso can model and query durable structured memory locally. Pro adds stronger PostgreSQL-native readiness, workspace governance, DDL lifecycle, and operator-grade batch paths.

## Runtime readiness

Use `ghostcrab_status` rather than static claims about the backend:

```json
{
  "tool": "ghostcrab_status",
  "arguments": {
    "agent_id": "agent:self"
  }
}
```

For Pro, important status fields include:

- `runtime.extensions_detected.pg_facets`
- `runtime.extensions_detected.pg_dgraph`
- `runtime.extensions_detected.pg_pragma`
- `runtime.extensions_detected.pg_mindbrain`
- `runtime.native_readiness`
- `runtime.capabilities`

For Perso, SQLite is expected and healthy when `database_kind` and `sqlite_backing_store` report the local backend as ready.
