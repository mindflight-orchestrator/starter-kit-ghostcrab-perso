# Semantic Graph

Relationships are first-class objects — typed, directional, traversable.

---

## Why foreign keys are not enough

A relational database can tell you that `task_id = 42` has `assigned_agent_id = 7`. It cannot tell you what that link *means* to an agent operating on it.

`blocks`, `depends_on`, and `requires` are all directed edges between entities. 

As SQL foreign keys, they're identical — a source id, a target id, a join. 

As semantic relations, they're completely different:

- **`blocks`** — active obstruction. The source cannot progress until the target resolves. Traversal direction: follow outbound to find what you are blocking; follow inbound (`blocked_by`) to find what is blocking you.
- **`depends_on`** — ordered dependency. The source needs the target to complete first, but does not halt it. Traversal: topological ordering for scheduling.
- **`requires`** — normative obligation. The source imposes a condition on the target's existence or state. Traversal: compliance coverage — *"are all required controls validated?"*

An agent that can only join on `task_id` cannot distinguish between these semantics. It must infer them from field names or documentation — which is exactly the kind of reconstruction work that projections eliminate.

In mindBrain, the relation type is the semantics. The graph engine traverses it accordingly.

## Declaring relations

Relations are declared on the source entity in `schema.ddl.yaml`:

```yaml
entities:

  Task:
    relations:
      - name: blocks               # relation name — verb, lowercase_snake
        target: Task               # target entity type
        direction: directed        # directed (default) or undirected
        inverse_name: blocked_by   # auto-generated reverse traversal
        cardinality: many-to-many  # one-to-one | one-to-many | many-to-many
        metadata:
          - name: since            # optional typed metadata on the edge itself
            type: datetime
          - name: reason
            type: string

      - name: assigned_to
        target: Agent
        direction: directed
        inverse_name: assignments
        cardinality: many-to-one   # many Tasks → one Agent

  Regulation:
    relations:
      - name: requires
        target: Control
        direction: directed
        inverse_name: required_by
        cardinality: one-to-many
        metadata:
          - name: article
            type: string           # e.g. "Art.32(1)(a)"
          - name: mandatory
            type: boolean
            default: true

  Control:
    relations:
      - name: validated_by
        target: Audit
        direction: directed
        inverse_name: validates
        cardinality: many-to-many
        metadata:
          - name: validated_at
            type: date
          - name: confidence
            type: float            # 0.0 – 1.0 coverage confidence
```

Every declared relation generates its inverse automatically. Declare `blocks`, traverse `blocked_by` immediately — no extra schema work.

## Auto-generated inverse traversal

When `schema load` runs, every relation with an `inverse_name` registers the reverse edge in the graph index:

```
✓ Relations compiled
    blocks          (Task → Task)       inverse: blocked_by
    assigned_to     (Task → Agent)      inverse: assignments
    requires        (Regulation → Control)  inverse: required_by
    validated_by    (Control → Audit)   inverse: validates
```

Inverse traversal is not a reverse query — it's a native graph operation. The engine walks the pre-indexed edge set in the declared direction. Depth, filters, and metadata access all work identically on inverse relations.

## Traversing the graph

**Depth 1 — direct neighbors:**

```bash
mindbrain traverse --from task-gdpr-art32 --relation blocks --depth 1
```

```json
{
  "start": "task-gdpr-art32",
  "relation": "blocks",
  "depth": 1,
  "nodes": [
    { "id": "task-gdpr-art35", "type": "Task", "status": "planned", "priority": "high" },
    { "id": "task-gdpr-art44", "type": "Task", "status": "planned", "priority": "medium" }
  ]
}
```

**Depth 2 — two hops:**

```bash
mindbrain traverse --from task-gdpr-art32 --relation blocks --depth 2
```

```json
{
  "start": "task-gdpr-art32",
  "relation": "blocks",
  "depth": 2,
  "tree": {
    "id": "task-gdpr-art32",
    "blocks": [
      {
        "id": "task-gdpr-art35",
        "blocks": [
          { "id": "task-gdpr-art44", "status": "planned" }
        ]
      }
    ]
  }
}
```

**Include edge metadata:**

```bash
mindbrain traverse   --from regulation-gdpr-art32   --relation requires   --depth 1   --include-metadata
```

```json
{
  "nodes": [
    {
      "id": "control-encryption-at-rest",
      "type": "Control",
      "status": "validated",
      "edge": { "article": "Art.32(1)(a)", "mandatory": true }
    },
    {
      "id": "control-access-logging",
      "type": "Control",
      "status": "pending",
      "edge": { "article": "Art.32(1)(b)", "mandatory": true }
    }
  ]
}
```

## Compound query — faceted collapse + graph traversal

Faceted filtering and graph traversal compose in a single operation:

```bash
mindbrain query   --type Task   --status blocked   --priority critical   --traverse assigned_to   --traverse-depth 1   --traverse blocks   --traverse-depth 2
```

The engine evaluates the faceted filter first (bitmap index, sub-millisecond), then runs the traversal on the resulting candidate set — not on the full graph. This keeps compound queries fast at scale: the traversal cost is proportional to the filtered set, not the total object count.

## Circular dependency detection

When a traversal encounters a cycle, mindBrain returns a structured warning rather than looping:

```bash
mindbrain traverse --from task-A --relation blocks --depth 10
```

```json
{
  "start": "task-A",
  "relation": "blocks",
  "depth_reached": 3,
  "cycle_detected": true,
  "cycle": {
    "path": ["task-A", "task-B", "task-C", "task-A"],
    "closing_edge": { "from": "task-C", "to": "task-A", "relation": "blocks" }
  },
  "warning": "Circular dependency detected at depth 3. Traversal halted. Path: task-A → task-B → task-C → task-A."
}
```

The `cycle` object gives the exact closing edge — the agent can surface it directly to a human without any interpretation work.

## Graph feeds projection

Graph traversal is not a standalone feature — it's the assembly mechanism behind projections.

When a projection declares `include_relations` and `depth`, the engine runs the equivalent traversal at cache-fill time:

```yaml
projections:
  compliance_cockpit:
    entity_type: Regulation
    depth: 2
    include_relations:
      - requires          # hop 1: Regulation → Control
      - validated_by      # hop 2: Control → Audit
    context_summary: true
    cache: true
```

At projection request time, the agent receives the pre-traversed graph — assembled at write time, served in ~2ms. The traversal happened; the agent just didn't pay for it at inference time.

Projections are cached traversals with a name.

## Real example — GDPR Art.32

A compliance agent needs a complete picture of GDPR Art.32: which controls it requires, and whether each has been audited.

**The graph:**

```
Regulation:gdpr-art32
    │
    ├─[requires]──► Control:ctrl-encryption    status: validated
    │                   └─[validated_by]──► Audit:audit-09    completed: 2026-03-15
    │
    ├─[requires]──► Control:ctrl-access-log    status: pending
    │                   └─[validated_by]──► (none)
    │
    ├─[requires]──► Control:ctrl-pseudonym     status: validated
    │                   └─[validated_by]──► Audit:audit-09    completed: 2026-03-15
    │
    └─[requires]──► Control:ctrl-breach-notif  status: pending
                        └─[validated_by]──► Audit:audit-11    scheduled: 2026-05-20
```

**One traversal command:**

```bash
mindbrain traverse   --from regulation-gdpr-art32   --relation requires   --depth 2   --include-metadata
```

Returns all four controls with their audit status, edge metadata (article reference, mandatory flag), and the audit dates — in a single operation.

**Or request the pre-assembled projection:**

```bash
mindbrain project --name compliance_cockpit --id regulation-gdpr-art32
```

Response: 2ms. The traversal already happened.
