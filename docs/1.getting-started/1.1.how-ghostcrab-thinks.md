# How GhostCrab Thinks

Most tools store what you tell them. GhostCrab models what you mean.

---

## The default assumption, and why it fails

Every memory tool makes an assumption about what "remembering" means.

The most common assumption is simple text recall: store a note, embed it, retrieve it later when a similar phrase appears. That works for recall. It does not work well for structural reasoning.

Consider a compliance agent that needs to answer: *"Which controls required by GDPR Art. 32 are not yet validated?"*

A text-retrieval system returns documents that mention GDPR Art. 32, controls, or validation. The agent then reads those documents and synthesizes an answer, spending tokens and risking a count that was never explicitly stored.

GhostCrab is built for a different shape of answer. If the graph contains a Regulation node linked to Control nodes with a typed `requires` edge, the agent can traverse the structure directly:

```json
{
  "tool": "ghostcrab_traverse",
  "arguments": {
    "start": "regulation-gdpr-art32",
    "edge_labels": ["requires"],
    "depth": 2,
    "workspace_id": "compliance-gdpr"
  }
}
```

`ghostcrab_traverse` is an extended tool, so an agent should discover its live schema first:

```json
{
  "tool": "ghostcrab_tool_search",
  "arguments": {
    "name_prefix": "ghostcrab_traverse",
    "include_input_schema": true,
    "limit": 5
  }
}
```

The difference is not retrieval speed. It is the shape of what is stored.

---

## Data without structure is noise

Raw data contains information. It does not necessarily contain meaning. Meaning requires structure: types, relations, states, and rules.

When an agent stores a fact without an ontology, it stores a string. When it stores the same fact with an ontology, it stores:

- an **entity** of a declared type, such as `Control`, `Regulation`, or `Task`
- a specific **position** in a typed dimension space, such as `status: pending` or `priority: high`
- typed **relations** to other entities, such as `requires`, `validated_by`, or `blocks`

The string is retrievable. The entity is navigable, filterable, traversable, and projectable. An agent that stores entities can answer structural questions such as *"what is blocked?"*, *"what is missing?"*, and *"what changed since the last session?"* without reconstructing everything from the transcript.

This is why modeling comes before writing. Without a model, every fact is just text. With a model, every fact has coordinates in a navigable space.

## MCP surfaces

GhostCrab Perso exposes a compact default MCP surface and a larger extended catalog.

The default visible tools are enough for most everyday work: status, search, count, remember, upsert, schema list/inspect, pack, project, modeling guidance, and tool search.

Extended tools such as `ghostcrab_workspace_create`, `ghostcrab_schema_register`, `ghostcrab_learn`, `ghostcrab_traverse`, and `ghostcrab_coverage` should be discovered before use:

```json
{
  "tool": "ghostcrab_tool_search",
  "arguments": {
    "name_prefix": "ghostcrab_workspace_",
    "include_input_schema": true,
    "limit": 10
  }
}
```

Documentation examples in this guide use live MCP argument shapes. If the catalog changes, trust `ghostcrab_tool_search` over older prose.

## What an ontology is, concretely

An ontology is not a philosophy exercise. In GhostCrab, it is the working model that tells the system what exists, what state can be tracked, and how things connect.

**1. Entity types**: the kinds of things in a domain.

```yaml
entities:
  - Regulation
  - Control
  - Audit
  - Task
  - Agent
```

**2. Dimensions**: the properties that make entities filterable and trackable.

```yaml
Control:
  dimensions:
    - name: status
      type: enum
      values: [pending, validated, failed]
    - name: domain
      type: enum
      values: [compliance, security, privacy]
    - name: mandatory
      type: boolean
```

**3. Relations**: the typed edges that connect entities.

```yaml
Regulation:
  relations:
    - name: requires
      target: Control

Control:
  relations:
    - name: validated_by
      target: Audit
```

Once the model is clear, every stored entity can be typed, positioned in dimension space, and connected by meaningful edges.

## Read before write

GhostCrab Perso is deliberately conservative on the first fuzzy onboarding turn. If a user says something broad like *"Help me track my team's onboarding tasks"*, the agent should not immediately create a workspace, register schemas, or write tasks.

The MCP-aligned pattern is:

1. Read runtime and catalog context when needed with `ghostcrab_status` and `ghostcrab_tool_search`.
2. Use `ghostcrab_modeling_guidance` for a fuzzy domain request.
3. Ask 2 to 4 high-impact clarification questions.
4. Write only after the user confirms the model.

The classic five-question intake is still useful as a teaching checklist:

1. What is the domain?
2. What are the main entity types?
3. What are the primary relations?
4. What state or workflow changes over time?
5. Are there existing data sources to import?

Those five questions are not a mandatory MCP first response. They are a human-readable way to explain the decisions that must eventually be clear.

## Model first, then data

A user asks: *"Help me track my team's onboarding tasks."*

Without modeling first, the agent might store three text notes:

```text
Alex needs to complete security training by Friday.
Jordan is blocked on access provisioning.
Onboarding checklist for new hires: ...
```

Those notes are searchable, but they are not reliably queryable by task status, assignee, or blocking relation.

With a model, the agent can store current-state records:

```json
{
  "tool": "ghostcrab_upsert",
  "arguments": {
    "schema_id": "ghostcrab:task",
    "workspace_id": "team-onboarding",
    "match": {
      "facets": {
        "record_id": "task:security-training"
      }
    },
    "set_content": "Alex needs to complete security training by Friday.",
    "set_facets": {
      "record_id": "task:security-training",
      "project": "team-onboarding",
      "owner": "alex",
      "status": "pending",
      "priority": "high"
    },
    "create_if_missing": true
  }
}
```

And it can add a typed graph edge:

```json
{
  "tool": "ghostcrab_learn",
  "arguments": {
    "workspace_id": "team-onboarding",
    "edge": {
      "source": "task:access-provisioning",
      "target": "task:security-training",
      "label": "blocks",
      "weight": 1
    }
  }
}
```

Later, the agent can answer the blocked-task question with a real MCP read:

```json
{
  "tool": "ghostcrab_search",
  "arguments": {
    "schema_id": "ghostcrab:task",
    "workspace_id": "team-onboarding",
    "filters": {
      "status": "blocked"
    },
    "limit": 20,
    "mode": "bm25"
  }
}
```

No fictional query language is required. The model gives the agent something exact to read.

---

## What this means for the agent

An agent operating with a declared model does not need to remember everything. It needs to store correctly and query directly.

The session transcript is not the knowledge. The graph and faceted records are the knowledge. The transcript is only the channel through which new facts arrive and get written.

This is the fundamental shift in how GhostCrab thinks: from *"store what was said"* to *"model what is true"*.
