# Your First Workspace

The intake conversation is done. The user has confirmed the model. Now the database can be built.

This guide uses GhostCrab Perso MCP payloads as exposed by the live tool catalog. When in doubt, inspect the current schema with `ghostcrab_tool_search` before calling an extended tool.

---

## What this sequence produces

At the end of this sequence:

- a named workspace exists
- schemas or ontology records describe the domain
- initial entities are ingested as typed current-state records
- graph relations connect the entities
- a compact projection is available for future sessions
- a checkpoint confirms the runtime state

Do not run these writes on the first fuzzy onboarding turn. Read first, model second, write only after confirmation.

---

## Step 0 - Discover the tools

The default MCP surface is compact. Workspace, schema, graph, traversal, and coverage tools are extended. Discover them before use:

```json
{
  "tool": "ghostcrab_tool_search",
  "arguments": {
    "name_prefix": "ghostcrab_workspace_",
    "include_input_schema": true,
    "limit": 10
  }
}
```

Repeat with prefixes such as `ghostcrab_schema_`, `ghostcrab_learn`, `ghostcrab_traverse`, and `ghostcrab_coverage` when you need those tools.

---

## Step 1 - Create the workspace

The workspace is the isolation scope. Use a stable lowercase id and a human-readable label.

```json
{
  "tool": "ghostcrab_workspace_create",
  "arguments": {
    "id": "compliance-gdpr",
    "label": "GDPR Compliance",
    "description": "Regulations, controls, audits, tasks, and agents for compliance tracking.",
    "created_by": "agent-compliance-01"
  }
}
```

If the workspace already exists, the tool is idempotent. Confirm the existing workspace instead of creating a duplicate:

```json
{
  "tool": "ghostcrab_workspace_list",
  "arguments": {
    "status": "active"
  }
}
```

## Step 2 - Register schemas

`ghostcrab_schema_register` is a freeze-level action. Use it only after the user confirms the model.

The live Perso shape uses a nested `definition` object. `workspace_id` is optional but recommended when the workspace is known.

```json
{
  "tool": "ghostcrab_schema_register",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "target": "facets",
    "definition": {
      "schema_id": "compliance:regulation",
      "description": "A regulation that requires one or more compliance controls.",
      "version": 1,
      "facets": {
        "record_id": "string",
        "title": "string",
        "status": "enum:active|deprecated",
        "jurisdiction": "enum:eu|us|global"
      }
    }
  }
}
```

```json
{
  "tool": "ghostcrab_schema_register",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "target": "facets",
    "definition": {
      "schema_id": "compliance:control",
      "description": "A compliance control tracked against regulations and audits.",
      "version": 1,
      "facets": {
        "record_id": "string",
        "title": "string",
        "status": "enum:pending|in_progress|validated|failed",
        "domain": "enum:compliance|security|privacy",
        "mandatory": "boolean"
      }
    }
  }
}
```

Register additional schemas for `Audit`, `Task`, and `Agent` only if they are part of the confirmed model. Keep the first version small enough to use immediately.

Verify registered schemas:

```json
{
  "tool": "ghostcrab_schema_list",
  "arguments": {
    "domain": "compliance",
    "summary_only": true
  }
}
```

## Step 3 - Inspect the workspace model

Before ingesting data, inspect the workspace:

```json
{
  "tool": "ghostcrab_workspace_inspect",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "include_columns": true,
    "include_relations": true
  }
}
```

Check that the workspace contains the schemas and semantics you expect. If the model is wrong, fix it before data exists.

---

## Step 4 - Ingest current-state records

Use `ghostcrab_upsert` for records whose state should change in place. Match by a stable facet such as `record_id`, then set content and facets.

```json
{
  "tool": "ghostcrab_upsert",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "schema_id": "compliance:regulation",
    "match": {
      "facets": {
        "record_id": "regulation:gdpr-art32"
      }
    },
    "set_content": "GDPR Art. 32 - Security of processing.",
    "set_facets": {
      "record_id": "regulation:gdpr-art32",
      "title": "GDPR Art. 32 - Security of processing",
      "status": "active",
      "jurisdiction": "eu"
    },
    "create_if_missing": true,
    "created_by": "agent-compliance-01"
  }
}
```

```json
{
  "tool": "ghostcrab_upsert",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "schema_id": "compliance:control",
    "match": {
      "facets": {
        "record_id": "control:encryption-at-rest"
      }
    },
    "set_content": "Encryption at rest control mapped to GDPR Art. 32.",
    "set_facets": {
      "record_id": "control:encryption-at-rest",
      "title": "Encryption at rest",
      "status": "pending",
      "domain": "security",
      "mandatory": true
    },
    "create_if_missing": true,
    "created_by": "agent-compliance-01"
  }
}
```

For CSV imports, validate the mapping first, then upsert rows in manageable batches. The MCP is the ontology and query surface; very high-throughput ingestion belongs on direct SQL or a dedicated import path.

---

## Step 5 - Link graph relations

Use `ghostcrab_learn` for durable structural relations. It accepts a `node` or an `edge`, with optional `workspace_id`.

```json
{
  "tool": "ghostcrab_learn",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "edge": {
      "source": "regulation:gdpr-art32",
      "target": "control:encryption-at-rest",
      "label": "requires",
      "weight": 1,
      "properties": {
        "source": "initial-import"
      }
    }
  }
}
```

To verify traversal, discover and call `ghostcrab_traverse`:

```json
{
  "tool": "ghostcrab_traverse",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "start": "regulation:gdpr-art32",
    "edge_labels": ["requires"],
    "direction": "outbound",
    "depth": 2
  }
}
```

---

## Step 6 - Check coverage

`ghostcrab_coverage` checks a domain against its registered ontology. Its required argument is `domain`.

```json
{
  "tool": "ghostcrab_coverage",
  "arguments": {
    "domain": "compliance",
    "workspace_id": "compliance-gdpr",
    "agent_id": "agent-compliance-01"
  }
}
```

Treat one low or empty coverage result as a diagnostic, not as proof that the whole workspace is empty. Check exact ids, schema ids, and relation labels before changing the model.

---

## Step 7 - Create the first projection

The projection is the agent's compact working context. Create it after the workspace has meaningful content.

`proj_type` must be one of `FACT`, `GOAL`, `STEP`, or `CONSTRAINT`.

```json
{
  "tool": "ghostcrab_project",
  "arguments": {
    "scope": "compliance-overview",
    "content": "GDPR compliance workspace initialized. Track controls by status and connect regulations to controls through requires edges.",
    "proj_type": "GOAL",
    "status": "active",
    "weight": 0.8,
    "activity_family": "compliance-audit",
    "agent_id": "agent-compliance-01",
    "provisional": false
  }
}
```

Resume the workspace later with `ghostcrab_pack`:

```json
{
  "tool": "ghostcrab_pack",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "scope": "compliance-overview",
    "query": "current compliance focus and pending controls",
    "agent_id": "agent-compliance-01",
    "limit": 15
  }
}
```

---

## Step 8 - Checkpoint runtime status

Use `ghostcrab_status` for the live runtime. Do not assume PostgreSQL native extension mode in the Perso starter path; SQLite is a normal default.

```json
{
  "tool": "ghostcrab_status",
  "arguments": {
    "agent_id": "agent-compliance-01"
  }
}
```

If extension flags appear, use the live key names:

```json
{
  "runtime": {
    "database_kind": "sqlite",
    "sqlite_backing_store": true,
    "extensions_detected": {
      "pg_facets": false,
      "pg_dgraph": false,
      "pg_pragma": false,
      "pg_mindbrain": true
    }
  }
}
```

The exact values depend on the backend. The key point is that the workspace, model, graph, and projection are now inspectable.
