# Your First Schema

In GhostCrab Perso, ontology structure lives in the GhostCrab database and is exposed through MCP tools and the `gcp brain` CLI.

This starter kit uses the current published CLI. It does not document a separate legacy YAML loader workflow.

---

## What "schema" means here

A schema or ontology profile describes the shape of a knowledge domain:

- entity types, such as `Task`, `Agent`, `Regulation`, or `Control`
- facets or dimensions, such as `status`, `priority`, `jurisdiction`, or `owner`
- relations, such as `blocks`, `assigned_to`, `requires`, or `validated_by`

The model gives the agent a stable contract for writes and queries. Without it, records drift into ad-hoc labels and text notes.

## Check the CLI

Use the package form when GhostCrab is not installed locally or globally:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp --help
```

The relevant commands for this guide are:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain workspace --help
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain schema --help
```

The workspace commands are:

```text
gcp brain workspace create [name]
gcp brain workspace list
```

The schema commands are:

```text
gcp brain schema list [--remote]
gcp brain schema pull <owner/name>
gcp brain schema show <owner/name>
gcp brain schema remove <owner/name>
```

`gcp ontologies <subcommand>` is an alias for the schema commands.

## Create a workspace

Create or register a local workspace:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain workspace create compliance-gdpr
```

List known workspaces and SQLite paths:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain workspace list
```

In the Perso starter path, the default backing store is a local SQLite file such as `./data/ghostcrab.sqlite`, depending on the current working directory and environment.

## Discover ontology profiles

List local profiles:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain schema list
```

List remote profiles from the registry:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain schema list --remote
```

Pull a profile:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain schema pull mindflight/mindbrain
```

Show a profile:

```bash
npx -y --package=@mindflight/ghostcrab-personal-mcp@latest gcp brain schema show mindflight/mindbrain
```

The exact profile names available depend on the configured registry.

## Register custom structure through MCP

For custom local schemas, use the live MCP surface after the user confirms the model.

First inspect the schema tool:

```json
{
  "tool": "ghostcrab_tool_search",
  "arguments": {
    "name_prefix": "ghostcrab_schema_",
    "include_input_schema": true,
    "limit": 10
  }
}
```

Then register a small schema:

```json
{
  "tool": "ghostcrab_schema_register",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "target": "facets",
    "definition": {
      "schema_id": "compliance:task",
      "description": "A task in a compliance workflow.",
      "version": 1,
      "facets": {
        "record_id": "string",
        "status": "enum:pending|in_progress|blocked|done",
        "priority": "enum:critical|high|medium|low",
        "owner": "string"
      }
    }
  }
}
```

This is a freeze-level write. Do not call it on the first fuzzy onboarding turn.

## Validate by reading

After registering or pulling structure, read it back:

```json
{
  "tool": "ghostcrab_schema_list",
  "arguments": {
    "domain": "compliance",
    "summary_only": true
  }
}
```

Inspect a single schema:

```json
{
  "tool": "ghostcrab_schema_inspect",
  "arguments": {
    "schema_id": "compliance:task"
  }
}
```

If the schema is workspace-specific, inspect workspace semantics as well:

```json
{
  "tool": "ghostcrab_workspace_inspect",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "include_columns": true,
    "include_relations": true
  }
}
```

## Glossary

**Entity**
A typed object in your domain, such as `Task`, `Agent`, `Regulation`, or `Document`.

**Facet / dimension**
A typed coordinate used for filtering, counting, grouping, or status tracking. Examples: `status`, `priority`, `jurisdiction`, `owner`.

**Relation**
A named edge between entities. Examples: `blocks`, `assigned_to`, `requires`, `validated_by`.

**Projection**
A compact working context assembled for an agent. In MCP, projections are managed with `ghostcrab_project` and read through `ghostcrab_pack`.
