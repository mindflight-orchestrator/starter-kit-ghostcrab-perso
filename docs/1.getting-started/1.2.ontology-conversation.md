# The Ontology Conversation

Before GhostCrab writes durable data, it needs enough structure to write the right thing.

---

## Why this conversation exists

A knowledge graph is only as useful as the model it is built on. Store entities without a schema, link them with ad-hoc relation names, or skip state dimensions, and the graph starts answering some queries while silently missing others.

The ontology conversation prevents that drift. It turns a fuzzy user request into a confirmed working model: workspace, entity types, states, relations, and ingestion plan.

The key rule is simple:

> Read before write. On a first fuzzy onboarding turn, do not create records, schemas, graph edges, or projections.

---

## When it triggers

The ontology conversation triggers on a first fuzzy request where the user describes a domain but has not specified a model:

- *"Help me track my team's onboarding tasks."*
- *"I want to build a brain for my SaaS product."*
- *"Set up a compliance tracking system for GDPR."*
- *"Model my CRM pipeline."*

A clear schema-first request does not need the same intake. If the user already gives a workspace id, entity types, fields, and relation names, the agent can validate and confirm that model before writing.

---

## The MCP-aligned intake

GhostCrab Perso exposes `ghostcrab_modeling_guidance` specifically for this moment:

```json
{
  "tool": "ghostcrab_modeling_guidance",
  "arguments": {
    "goal": "Track compliance work for GDPR across my team."
  }
}
```

The first response should ask 2 to 4 high-impact clarification questions, not an exact fixed script. Good questions resolve decisions that affect the model:

- What domain or project name should scope this workspace?
- What are the main things to track?
- Which states change over time?
- What sources or existing records should be imported?

The classic five-question checklist is still useful as a teaching version of the same intake:

1. What is the domain?
2. What are the main entity types?
3. What are the primary relations between them?
4. What state or workflow needs to be tracked over time?
5. Are there existing data sources to import?

Use that checklist when it helps the user think. Do not present it as a server mandate, and do not write before the user confirms the model.

---

## MCP surfaces

The default GhostCrab Perso MCP list is intentionally compact. It includes `ghostcrab_modeling_guidance`, `ghostcrab_search`, `ghostcrab_upsert`, `ghostcrab_pack`, `ghostcrab_project`, and `ghostcrab_tool_search`.

Workspace, schema, graph, traversal, and coverage operations are extended tools. Discover them before documenting or calling them:

```json
{
  "tool": "ghostcrab_tool_search",
  "arguments": {
    "name_prefix": "ghostcrab_schema_",
    "include_input_schema": true,
    "limit": 10
  }
}
```

---

## What each question determines

| Topic | What it determines |
|---|---|
| Domain | Workspace id, label, and naming conventions |
| Entity types | Schema ids or ontology entity types |
| Relations | Graph edge labels and direction |
| State / workflow | Facets, status values, projection focus |
| Existing data | Import order, mapping, and validation strategy |

No topic is decorative. Each one prevents a later class of query failure.

## A real intake

**User:** *"I want to track compliance work for GDPR across my team."*

**Agent reads guidance and asks a focused intake.**

```text
Before I model this, I need to pin down the structure.

1. Should the workspace be scoped to GDPR only, or GDPR plus ISO 27001?
2. What should we track as first-class entities: Regulations, Controls, Audits, Tasks, Agents?
3. Which states matter most for day-to-day work: control validation, task progress, ownership, or blockers?
4. Are we starting empty, or importing an existing spreadsheet?

I will wait for your answers before writing to GhostCrab.
```

**User answers:**

```text
GDPR and ISO 27001. Track Regulation, Control, Audit, Task, and Agent.
Relations: Regulation requires Control. Control validated_by Audit. Task assigned_to Agent. Task blocks Task.
Track Control.status and Task.status. I have a CSV with about 80 controls.
```

**Agent confirms the model before writing:**

```text
I will create workspace `compliance-gdpr`, register schemas for Regulation, Control, Audit, Task, and Agent, then ingest the CSV after mapping its columns.

Key relations:
Regulation requires Control
Control validated_by Audit
Task assigned_to Agent
Task blocks Task

Key states:
Control.status: pending | in_progress | validated | failed
Task.status: pending | blocked | done
```

Only after confirmation should the agent call write tools.

## Valid write sequence after confirmation

Create the workspace with the extended workspace tool:

```json
{
  "tool": "ghostcrab_workspace_create",
  "arguments": {
    "id": "compliance-gdpr",
    "label": "GDPR Compliance",
    "description": "Regulations, controls, audits, tasks, and agents for compliance tracking."
  }
}
```

Register a schema with a nested definition:

```json
{
  "tool": "ghostcrab_schema_register",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "target": "facets",
    "definition": {
      "schema_id": "compliance:control",
      "description": "A compliance control tracked against regulations and audits.",
      "version": 1,
      "facets": {
        "record_id": "string",
        "status": "enum:pending|in_progress|validated|failed",
        "domain": "string",
        "mandatory": "boolean"
      }
    }
  }
}
```

Write a current-state record with `ghostcrab_upsert`:

```json
{
  "tool": "ghostcrab_upsert",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "schema_id": "compliance:control",
    "match": {
      "facets": {
        "record_id": "control:encryption-at-rest"
      }
    },
    "set_content": "Encryption at rest control for GDPR Art. 32.",
    "set_facets": {
      "record_id": "control:encryption-at-rest",
      "status": "pending",
      "domain": "security",
      "mandatory": true
    },
    "create_if_missing": true
  }
}
```

Link graph structure with `ghostcrab_learn`:

```json
{
  "tool": "ghostcrab_learn",
  "arguments": {
    "workspace_id": "compliance-gdpr",
    "edge": {
      "source": "regulation:gdpr-art32",
      "target": "control:encryption-at-rest",
      "label": "requires",
      "weight": 1
    }
  }
}
```

Create a compact working projection:

```json
{
  "tool": "ghostcrab_project",
  "arguments": {
    "scope": "compliance-overview",
    "content": "GDPR compliance workspace initialized. Track Control.status and Task.status, with Regulation requires Control and Task blocks Task relations.",
    "proj_type": "GOAL",
    "status": "active",
    "agent_id": "agent-compliance-01",
    "provisional": false
  }
}
```

Check coverage by domain:

```json
{
  "tool": "ghostcrab_coverage",
  "arguments": {
    "domain": "compliance",
    "workspace_id": "compliance-gdpr",
    "agent_id": "agent-compliance-01"
  }
}
```

## Closing checkpoint

After setup, give the user a concrete status summary. Runtime flags vary by backend, so do not hard-code PostgreSQL native mode as the expected result.

```text
Your GhostCrab workspace is live.

Workspace:  compliance-gdpr
Schemas:    Regulation, Control, Audit, Task, Agent
Graph:      Controls linked to Regulations through `requires`
Projection: compliance-overview
Runtime:    SQLite Perso backend unless your status response says otherwise

From now on:
- I read from GhostCrab before answering operational questions.
- I write only confirmed new durable facts.
- I update the active projection when the working focus changes.
```

Use `ghostcrab_status` for the actual runtime:

```json
{
  "tool": "ghostcrab_status",
  "arguments": {
    "agent_id": "agent-compliance-01"
  }
}
```

If extension flags are shown, use the live key names: `pg_facets`, `pg_dgraph`, and `pg_pragma`.
